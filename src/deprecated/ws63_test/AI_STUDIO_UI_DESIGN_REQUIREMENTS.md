# AI Studio UI 设计需求梳理

本文档用于给 UI/UX 设计师快速理解 `ai_studio` 当前必须承载的完整能力边界。

目标不是讨论“代码怎么写”，而是把下面几件事讲清楚：

- `ai_studio` 这个产品到底要解决什么问题
- 现在必须具备哪些功能，哪些不能在改版中丢失
- UI 上哪些信息必须清晰可见
- 设计时有哪些硬约束
- 当前体验里最需要被解决的具体问题是什么

---

## 1. 产品定位

`ai_studio` 是 `ws63_test` 的 PC 端 AI 上位机，不是通用绘图软件，也不是单纯的串口助手。

它的真实定位是：

- 面向比赛展示、课堂演示、联调验证的“创作到执行”一体化控制台
- 把 `AI 生图 / 本地图导入 -> OpenCV 轮廓提取 -> G-Code 生成 -> 串口或 WiFi 下发 -> WS63 真机执行` 这条链路完整打通
- 既要能给评委看闭环，也要能给操作者稳定地把任务发到设备

一句话概括：

`ai_studio` 不是单一功能界面，而是一个“面向激光打标比赛演示的任务工作台”。

---

## 2. 设计目标

本轮 UI 改版必须同时满足下面 4 个目标：

1. 让第一次接触项目的人也能快速看懂主流程
2. 让演示者在比赛现场少走弯路、少误操作
3. 让关键状态始终可见，不靠翻日志猜当前阶段
4. 让界面在功能增加后不再相互挤压、裁切、抢空间

当前最需要解决的问题：

- 模块之间层层嵌套，导致一多就挤压
- 关键状态、关键动作、关键反馈没有形成稳定的信息层级
- 预览区、参数区、执行区、历史区经常互相抢空间
- 日志和任务状态有价值，但不该吞掉主要工作区域
- 一些控件容易误触，例如滚轮误改参数

---

## 3. 核心闭环

这是 `ai_studio` 必须始终围绕的主线，UI 设计不能把这个闭环打散：

```text
创意输入
  -> AI 生图 或 本地图导入
  -> 轮廓提取
  -> G-Code 生成
  -> 设备连接确认
  -> 下发执行
  -> 实时状态 / 结果反馈 / 历史留档
```

UI 设计上必须让用户一眼知道：

- 我现在处于哪一步
- 下一步该做什么
- 当前任务是否成功推进
- 如果失败，失败在什么阶段

---

## 4. 必须具备的全部功能

下面这些能力是当前 `ai_studio` 必须支持的，不是“可选增强”。

### 4.1 设备连接与通路选择

必须支持：

- `串口 UART`
- `WiFi TCP`

WiFi 又必须区分两种工作流：

- `WiFi SoftAP`
- `WiFi STA`

连接相关能力必须包括：

- 选择连接方式
- 快捷切换预设：`串口模式 / WiFi SoftAP / WiFi STA`
- 串口列表刷新
- 串口选择
- 波特率选择
- WiFi 方案选择
- WiFi IP 输入
- WiFi TCP 端口输入
- 连接设备
- 断开设备

连接相关信息必须能被清楚展示：

- 当前采用的是哪种链路
- 当前连接目标是什么
- 当前是否已经连接
- 当前 WiFi 工作流是 `SoftAP` 还是 `STA`

### 4.2 实时设备状态展示

界面必须能展示设备实时状态，当前真实可用数据包括：

- 连接方式
- 设备状态：如 `Idle / Run`
- 实时坐标：`X / Y`
- 当前进给：`F`
- 当前功率：`S`
- 当前连接目标
- 状态刷新时间

在 WiFi 模式下，最好还能展示：

- WiFi 模式
- `SLE` 是否就绪

这部分对比赛展示非常重要，因为它能证明：

- 上位机不是“假流程”
- 真机链路确实已经接通
- 当前设备确实在响应

### 4.3 AI 创作能力

必须支持：

- 输入 Prompt
- 使用比赛模板快速填入 Prompt
- 点击生成 AI 图像

当前 AI 相关事实：

- 使用 Google `Imagen 4 Fast`
- 生成结果会自动保存到本地
- 生成成功后应继续进入后续处理链路

从 UI 角度必须考虑：

- Prompt 输入区不能太局促
- 模板选择应该足够快
- “生成中”状态要明显
- 网络失败、超时、API Key 问题要有清晰反馈

### 4.4 本地图导入能力

必须支持：

- 从本地选择图片文件
- 兼容常见图片格式
- 导入后自动进入轮廓提取流程

这条链路非常重要，因为它是：

- 无外网时的降级方案
- 比赛现场的兜底方案
- 联调时最快的验证路径

### 4.5 轮廓提取与图像处理

必须支持：

- 图像预处理
- OpenCV 轮廓提取
- 轮廓归一化
- 居中排版
- 输出轮廓预览图

界面上至少要让用户知道：

- 原图是什么
- 轮廓结果是什么
- 当前轮廓提取是否成功

这一步是整个系统的核心解释点之一，因为 AI 图片不能直接下发，必须先转成可雕刻轨迹。

### 4.6 G-Code 生成

必须支持：

- 根据轮廓生成 G-Code
- 根据当前宽度、高度、进给、功率参数生成
- 导出 `.gcode`
- 生成边框预览 G-Code

约束必须明确：

- 当前工作区默认为 `100mm x 100mm`
- G-Code 输出保持第一象限正坐标
- 生成结果必须兼容当前 WS63 固件链路

用户需要能清楚看到：

- 当前参数是什么
- G-Code 是否已经生成
- 预览路径与真实雕刻路径的区别

### 4.7 设备执行与安全控制

必须支持：

- 下发 G-Code 到设备
- 红光边框预览
- 开始激光雕刻
- 紧急停止

其中“急停”是绝对不能弱化的重点能力：

- 串口模式急停：`M5 + !`
- WiFi 模式急停：`!`
- 急停后要有明确反馈

设计上必须把“急停”做成高优先级、高识别度动作，不应埋得很深。

### 4.8 一键比赛流程

必须支持“一键比赛流程”：

```text
AI 生图 -> 轮廓提取 -> G-Code 生成 -> 发送到当前设备
```

这不是一个普通按钮，而是比赛展示中的核心卖点。

UI 需要清楚表达：

- 这是完整闭环动作
- 它依赖当前设备已连接
- 它在执行时会经历多个阶段
- 任一阶段失败都要能解释失败位置

### 4.9 任务状态机与进度展示

当前任务状态机已经存在，设计上必须完整承接：

- `等待任务`
- `AI 生成中`
- `轮廓提取中`
- `G-Code 生成中`
- `下发中`
- `完成`
- `失败`

这部分必须可视化。

建议设计师明确考虑：

- 当前阶段高亮
- 已完成阶段
- 失败阶段
- 执行百分比
- 执行中文说明

这是解释“系统正在做什么”的核心模块。

### 4.10 历史任务与产物留档

必须支持：

- 查看历史任务
- 查看任务详情
- 复用历史任务
- 打开任务产物目录

每个任务当前会自动留档：

- 原图
- 轮廓图
- G-Code
- 任务元数据

历史区的价值不是“有就行”，而是：

- 比赛复盘
- 成果复用
- 演示时快速切回成功案例
- 失败问题追溯

### 4.11 日志与错误反馈

必须支持：

- 系统消息板
- 串口 / WiFi 收发日志
- G-Code 预览日志
- 任务完成 / 失败提示

并且错误信息不能只给技术原文，还要有可读化结果，例如：

- 超时
- API Key 问题
- WiFi 连接失败
- 串口异常
- 轮廓提取失败
- 下位机返回错误
- 急停触发

日志区必须存在，但不应成为主视觉中心。

---

## 5. 关键使用场景

设计师至少要围绕下面 5 个真实场景设计。

### 场景 A：比赛现场完整演示

流程：

- 连接设备
- 选择模板
- 输入或调整 Prompt
- 点击一键比赛流程
- 观察 AI 生图、轮廓、任务状态、设备执行
- 必要时急停

这是最重要的展示场景。

### 场景 B：无外网时的本地图兜底

流程：

- 连接设备
- 导入本地图片
- 轮廓提取
- 生成 G-Code
- 发送执行

这是比赛现场最重要的“保底路径”。

### 场景 C：纯联调

流程：

- 先只连设备
- 看连接是否成功
- 看实时状态是否在刷新
- 发红光边框预览
- 发一小段雕刻任务

这时 UI 重点不是漂亮，而是信息稳定、动作直接。

### 场景 D：历史成果复用

流程：

- 从历史记录中找到成功任务
- 查看详情
- 复用参数和产物
- 重新发送

这时 UI 应该支持“快速恢复一个可演示状态”。

### 场景 E：问题定位 / 失败恢复

流程：

- 某一步失败
- 快速看出失败在哪一段
- 看可读化错误
- 看日志
- 决定重试、切换链路或改走本地图方案

这时 UI 应尽量减少认知负担。

---

## 6. 设计时不能违背的硬约束

这部分非常重要，属于 UI 设计必须服从的产品约束。

### 6.1 这是“比赛工作台”，不是纯展示海报

必须兼顾：

- 可讲解
- 可操作
- 可排错

不能只追求酷炫，而牺牲操作效率和状态清晰度。

### 6.2 关键动作不能被埋太深

下列动作必须在主工作区可快速触达：

- 连接设备
- AI 生成
- 导入本地图片
- 生成 G-Code
- 边框预览
- 开始雕刻
- 急停

### 6.3 不能依赖隐藏的信息才能判断系统状态

以下状态必须在主界面直接可见：

- 当前连接方式
- 是否已连接
- 当前任务阶段
- 当前是否正在执行
- 当前设备实时状态
- 当前主要失败原因

### 6.4 布局必须抗挤压

当前痛点之一就是模块互相挤压，所以新设计必须考虑：

- 功能继续增加后仍然不崩
- 日志区不挤压主流程区
- 预览区必须保有足够空间
- 参数区和执行区不能因为窗口高度变化而裁切严重

建议设计师优先考虑：

- `顶部状态条 + 中部多栏工作台 + 底部日志`
- 或其他同等清晰的桌面工作台布局

### 6.5 必须适配 Windows 桌面使用习惯

当前实际运行环境包含：

- Windows 11
- PowerShell 启动
- 常见桌面显示器
- Windows 缩放比例可能为 `125% / 150%`

因此设计上必须尽量避免：

- 字体裁切
- 按钮高度不够
- 多层嵌套导致滚动冲突

### 6.6 必须考虑误操作防护

当前已经明确遇到过的误操作问题：

- 鼠标滚轮误改参数
- 滚轮误改端口 / 波特率

所以设计必须天然考虑：

- 关键参数改动要更克制
- 高频危险动作需要更明确区分
- 急停不能误触，但也不能难找

### 6.7 不能假设永远有网络

设计必须承认这件事：

- `SoftAP` 下可能没有外网
- AI 生图可能失败
- 本地图导入是正式兜底路径，不是边角功能

因此“本地图导入”在信息架构上必须和“AI 生图”同级，而不是藏起来。

---

## 7. 设计上必须清晰表达的业务区别

UI 里必须能讲清楚下面这些区别。

### 7.1 `SoftAP` 与 `STA` 的区别

`SoftAP`：

- 适合现场直连
- 控制链路最短
- 可能不方便同时在线生图

`STA`：

- 适合和路由器 / 手机热点共网
- 更适合在线 AI 生图 + WiFi 下发

这两者不能只靠技术名词区分，最好有场景化说明。

### 7.2 `AI 生图` 与 `本地图导入` 的区别

`AI 生图`：

- 更适合展示“AI + 控制”闭环
- 依赖云端接口和网络

`本地图导入`：

- 更稳
- 更适合现场兜底和快速验证

### 7.3 `边框预览` 与 `正式雕刻` 的区别

边框预览：

- 用于确认范围
- 不是正式雕刻任务

正式雕刻：

- 真正执行轨迹
- 会用到激光功率与运动参数

### 7.4 `任务状态` 与 `设备状态` 的区别

任务状态是软件工作流：

- AI 生成中
- 轮廓提取中
- G-Code 生成中
- 下发中

设备状态是硬件当前反馈：

- Idle
- Run
- 实时坐标

这两个概念不能混在一起。

---

## 8. 建议 UI 信息架构

这部分不是最终设计稿，只是建议设计师优先围绕的结构。

### 8.1 顶部全局状态区

建议始终展示：

- 产品名
- 当前连接方式
- 设备在线状态
- WiFi / SLE 状态
- 实时坐标
- 当前执行参数摘要

### 8.2 左侧：输入与配置

建议收纳：

- 设备连接
- AI 模板与 Prompt
- 本地图导入
- 工艺参数

### 8.3 中间：主舞台

建议以大面积视觉区域展示：

- 原图
- 轮廓图
- 主流程箭头或阶段说明

这是最适合给评委看的区域。

### 8.4 右侧：任务执行区

建议展示：

- 任务状态机
- 当前进度
- 执行动作按钮
- 历史任务快速入口

### 8.5 底部：日志与错误

建议保留，但降低视觉权重：

- 日志滚动区
- 最近收发
- 最近错误

---

## 9. 必须覆盖的状态与空态

设计稿不能只画“理想成功态”，至少还要考虑这些状态：

- 未连接设备
- 已连接但未开始任务
- AI 生成中
- AI 生成失败
- 轮廓提取失败
- G-Code 生成成功
- 设备下发中
- 设备执行中
- 任务完成
- 任务失败
- 急停触发后
- 无历史记录
- 有历史记录但当前未选中

---

## 10. 业务约束与协议边界

设计师不需要深入固件，但必须知道这些边界：

- 下位机接受的是文本 G-Code
- 当前链路支持 `G0 / G1 / G90 / M3 / M5 / S / F`
- 每行执行后返回 `ok`
- `?` 可查询实时状态
- WiFi 采用 TCP 文本入口
- 当前板端只允许一个上游主机有效控制

这意味着 UI 不应该暗示：

- 多设备并发控制
- 多上位机同时接入
- 多任务后台并行下发

---

## 11. 当前版本设计最应重点提升的方向

如果交给设计师优先攻坚，我建议重点是下面 6 件事：

1. 重新建立信息层级，让“状态 / 预览 / 动作”分区明确。
2. 做出更强的工作台结构，解决模块互相挤压。
3. 把 `SoftAP / STA / 本地图 / AI 生图` 的业务差异讲清楚。
4. 让一键比赛流程真正成为主角，而不是普通按钮。
5. 让历史任务变成有价值的“成果复用区”，不是附属列表。
6. 让急停、失败反馈、当前阶段更容易被一眼识别。

---

## 12. 优先级建议

如果设计师需要区分优先级，可以按下面分。

### P0：绝对不能丢

- 设备连接
- AI 生图
- 本地图导入
- 轮廓提取结果展示
- G-Code 生成
- 红光边框预览
- 开始雕刻
- 急停
- 任务状态展示
- 实时设备状态
- 历史任务
- 日志与错误反馈

### P1：强烈建议重点做好的体验层

- 一键比赛流程的主视觉入口
- `SoftAP / STA` 场景化说明
- 连接后实时状态区
- 失败阶段的明确反馈
- 历史任务的复用效率

### P2：可以后续再增强

- 更强的任务阶段动画
- 更精致的作品流转展示
- 更完整的成果图库式历史界面
- 更适合答辩的大屏展示模式

---

## 13. 给设计师的最终一句话

请把 `ai_studio` 设计成一个：

`能让评委一眼看懂闭环、让操作者不容易犯错、让演示者在现场能稳稳完成“AI 到真机打标”全过程的桌面工作台。`

不是做成普通表单页，也不是做成只好看但难操作的海报式界面。
