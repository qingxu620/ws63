# WS63 项目能力现状、可行性评估与推进计划

更新时间：`2026-04-19`
适用目录：`ws63_test/`

这份文档面向项目成员、指导老师和答辩场景，核心目标有 3 个：

1. 说明当前项目已经实现了哪些功能
2. 评估既定提升方向中哪些适合落地、哪些难度较高
3. 给出一份按阶段推进的实际计划

---

## 1. 一句话项目定位

当前项目已经完成了一个基于 `WS63 + LiteOS` 的双板激光打标控制样机，主链路为：

`上位机 / AI 设计端 -> 发射板 -> SLE -> 接收板 -> DAC/PWM -> 激光执行`

它已经不再是单点功能验证，而是具备了：

- 多端接入
- 无线控制
- AI 设计雏形
- 安全停机基础
- 自动化测试与文档支撑

的完整工程基础。

## 2. 当前已经实现了哪些功能

为了更清楚地区分成熟度，当前能力分为三类：

- `已实现并已验证`
- `已实现但仍需补充验收`
- `已具备基础，仍需继续打磨`

## 2.1 已实现并已验证的功能

### A. 双板主控制链路

已完成：

- 基于 `WS63 + LiteOS` 的双板分工架构
- 发射板负责 `UART / WiFi / G-Code / SLE Client`
- 接收板负责 `SLE Server / 命令队列 / 插补执行 / DAC / PWM / 安全监控`
- `上位机 -> 发射板 -> 接收板 -> 执行输出` 主链路已跑通

说明：

- 这是整个项目最核心、最确定的实现成果
- 后续所有扩展都建立在这条链路之上

### B. UART 文本 G-Code 控制入口

已完成：

- 发射板 `UART1` 文本命令接收
- `G0/G1/G90/G91/G92/M3/M4/M5/S/F/?/$I/$G` 等常见指令处理
- Grbl 风格的 `ok / error / status` 反馈

说明：

- 串口链路已经是稳定基线
- 适合做本地联调和稳定性回归

### C. SLE 板间无线实时控制

已完成：

- 发射板 `SLE Client`
- 接收板 `SLE Server`
- 命令发送、状态回传、ACK 与队列协同
- 基础流控与心跳保活

说明：

- 这部分直接体现了项目的无线边缘控制能力
- 是后续讲“多 WS63 节点协同”的基础

### D. 接收板运动与激光执行层

已完成：

- 命令队列
- 线性插补
- `DAC8562` 输出振镜 `X/Y`
- `PWM` 控制激光功率
- 状态回传与位置反馈

说明：

- 这部分已经完成了“从命令到真实执行”的闭环

### E. 安全监控与真急停基础

已完成：

- SLE 超时停光保护
- 本轮新增真急停命令链路
- `! / $STOP / M112` 可统一触发急停
- 接收板收到急停后可立即：
  - 关闭激光
  - 清空命令队列
  - 打断当前运动

说明：

- 这一项非常关键
- 它不仅是安全能力，也是后续 `BLE / 网页 / 安全节点 / 云端联动` 的基础底座

## 3. 提升方向可行性分析

这里把你提出的提升方向分成三层：

- `第一层：当前赛季适合落地`
- `第二层：可以实现，但依赖新增硬件与更多联调`
- `第三层：实现难度高，不建议当前作为主线`

## 3.1 第一层：当前赛季适合落地

这些方向与现有代码衔接最好，比赛收益也最高。

### A. AIGC 云边协同打标

结论：

`强烈推荐，且实现可行性高。`

原因：

- `ai_studio` 已有 AI 生图、图像处理、G-Code 生成、设备下发能力
- 当前缺的主要是“任务化”和“展示化”，不是从零开始开发

更准确地说，这条路线当前已经完成了“70% 的主干”，剩下的是把它从“能跑”升级为“像一个正式比赛能力”。

当前已经具备的底座：

- `ai_studio` 可直接调用云端 AI 生图接口
- AI 结果可自动进入 OpenCV 轮廓提取
- 已能生成当前板端兼容的 G-Code
- 已能通过串口或 WiFi 下发到 WS63 发射板
- 已有“一键比赛流程”基础闭环

这意味着：

`你们现在不需要从零重做云平台，只需要把现有 PC 端 AI 工作流产品化、任务化、展示化。`

推荐的目标定义：

- 第一阶段目标不是“做一个完整网站”
- 第一阶段目标是“让评委清楚看到：一句 Prompt -> AI 生成图案 -> 自动转轨迹 -> WS63 下发执行”

换句话说，当前赛季最推荐的实现形式其实是：

`PC 端在线 AI 设计中枢 + 局域网/串口边缘执行`

这已经足够构成“云边协同”故事：

- 云端负责内容生成
- 边缘端负责图像处理、轨迹生成与设备执行
- WS63 负责无线控制与实时打标

### 推荐架构

建议按下面的最小闭环理解这条能力：

```text
用户 Prompt / 模板选择
    ->
ai_studio 调用云端 AI 生图 API
    ->
本地缓存 AI 原图与元数据
    ->
OpenCV 轮廓提取 / G-Code 生成
    ->
串口或 WiFi 下发到 WS63
    ->
WS63 执行并返回状态
    ->
ai_studio 展示任务进度 / 成功失败结果
```

这条链路的好处是：

- 不需要额外先搭服务器
- 与当前代码最贴近
- 比赛演示更稳定
- 后续也容易逐步升级到 Web 或 MQTT 方案

### 当前赛季建议做到的最小版本

如果按“最小可交付比赛能力”来定义，建议至少做到下面 6 点：

1. 固化 `2~3` 套高成功率 Prompt 模板。
2. 每次 AI 生成后自动保存原图、轮廓图、G-Code 和任务时间戳。
3. 在界面中显示清晰的任务状态：
   `等待生成 -> AI 生成中 -> 轮廓提取中 -> G-Code 生成中 -> 下发中 -> 执行完成 / 失败`
4. 在发送完成后记录任务结果：
   - 使用了哪个模板
   - 生成图像路径
   - 是否成功发送
   - 是否完成执行
5. 在主界面提供“最近任务”或“历史记录”入口，便于现场快速复用上一次成功作品。
6. 把失败原因做成可读提示，而不是只在日志里堆异常文本。

只要把这 6 点补齐，这条线就已经非常能打。

### 还需要补的内容

建议分成 4 类来看：

#### 1. Prompt 与模板层

- 固化比赛模板
- 区分“展示优先”和“雕刻优先”两类提示词
- 给不同模板配默认尺寸、功率、速度建议

#### 2. 任务管理层

- 历史任务与缓存
- 最近成功作品快速复用
- 任务状态机与阶段进度显示
- 任务结果留档

#### 3. 展示交互层

- 更顺畅的 AI -> 轮廓 -> G-Code -> 下发展示流程
- 更清晰的步骤提示和当前阶段标记
- 更适合答辩的状态面板和日志文案

#### 4. 后续扩展层

- 可选接入 MQTT 做任务状态遥测
- 可选把 Prompt 与历史任务迁到轻量后端
- 可选做 Web 端提交任务、PC 端执行的云边版架构

### 推荐实施顺序

建议按下面顺序推进，不要同时开太多战线：

1. 先做模板固化。
2. 再做历史任务与缓存。
3. 再做任务状态面板。
4. 再做失败原因和执行结果留档。
5. 最后才考虑 MQTT 或 Web 化扩展。

原因很简单：

`前四项直接提升比赛成功率和演示效果，第五项更偏加分项。`

### 风险点与规避建议

这条路线虽然可行性高，但也有 4 个要提前控制的风险：

#### A. AI 出图不稳定

风险：

- 同一个 Prompt 每次结果可能差异较大
- 某些图适合展示，但不适合轮廓提取

规避：

- 固定模板
- 固定关键词后缀
- 优先沉淀高成功率风格，而不是追求“什么都能画”

#### B. 云端接口依赖网络

风险：

- 比赛现场网络不稳定会直接影响 AI 生图

规避：

- 准备若干已缓存成功样例
- 保留“本地图导入”作为降级路径
- 现场优先用 `STA` 或双网方案，不把 SoftAP 作为在线生图主路径

#### C. 任务链路太长，评委不容易理解

风险：

- 如果界面没有状态提示，评委只会看到“等很久”

规避：

- 必须做任务状态机
- 每个阶段给出明确提示文案
- 尽量让一条完整任务在可接受时间内闭环

#### D. 日志太工程化，不利于展示

风险：

- 只有开发日志，缺少“用户可读反馈”

规避：

- 给界面增加面向演示的状态描述
- 开发日志保留在下面，展示提示放在上面

### 建议验收标准

这条能力建议至少满足下面这些验收标准：

- 用户选择一个模板并输入 Prompt 后，可在 `3~5` 步内完成打标任务下发。
- 任意一次成功任务都能在本地找到：
  - AI 原图
  - 轮廓图
  - G-Code
  - 任务日志
- 任务状态在界面上可见，且失败时能给出可读原因。
- 在没有外网时，仍可切换到“本地图导入 -> G-Code -> 发送”的降级路径。

难度判断：

`中低`

推荐指数：

`★★★★★`

## 3.2 第二层：可以实现，但依赖新增硬件与更多联调

这些方向不是不能做，而是会显著增加系统复杂度。

### A. 星闪 SLE 无线示教器

结论：

`可以做，但复杂度高于 BLE 手机轻控制。`

原因：

- 当前系统已经使用一条正式 `SLE` 主链路
- 如果再加一个示教器节点，会涉及第三节点协议与角色设计

建议：

- 第一版只做 `jog / 开始 / 停止 / 急停`
- 不承载完整 G-Code 流

难度判断：

`中高`

推荐指数：

`★★★☆☆`

### B. NFC 材质识别与工艺推荐

结论：

`可以做，展示效果不错，但需要明确挂载方案。`

原因：

- 逻辑上很好实现
- 但 `RC522` 不能简单直接挂到当前接收板 `SPI0`，因为 `SPI0` 已被 `DAC8562` 占用

建议：

- 优先挂在发射板
- 或单独挂到感知节点

第一版建议：

- 读 UID
- 查本地材质参数表
- 自动填入推荐功率和速度

难度判断：

`中`

推荐指数：

`★★★★☆`

### C. RS485 激光三角位移传感器

结论：

`值得做，但应先作为辅助测高，不要直接做实时主闭环。`

还需要补：

- `RS485 / Modbus RTU` 驱动
- 数据滤波
- 标定
- 零点建立流程

建议顺序：

1. 单次读值
2. 稳定滤波
3. 上电测高
4. 单点自动对焦

难度判断：

`中高`

推荐指数：

`★★★☆☆`

### D. Z 轴自动对焦执行器

结论：

`可以做，但它本质上已经是一个新子系统。`

还需要补：

- Z 轴通信协议
- 回零与当前位置管理
- 加工前自动对焦流程
- 急停联动策略

建议：

- 第一版只做绝对高度设定、回零、当前位置读取
- 不要一上来就做加工中高频跟随

难度判断：

`高`

推荐指数：

`★★★☆☆`

## 3.3 第三层：实现难度高，不建议当前作为主线

### A. 无场镜微型振镜 + 3D 动态焦点补偿

结论：

`理论上可做，但当前阶段不适合作为主攻方向。`

主要原因：

1. 当前协议只有 `X/Y/Feed/Power`，没有 `Z` 轴同步控制字段
2. 当前插补器是二维执行模型，没有 `XY + Z` 联动规划
3. 动态补偿不仅是控制问题，还涉及机械结构、传感精度、标定模型和光学误差
4. 机械 Z 补偿不能完全替代场镜和光学校正

更现实的路线应该是：

- 先做单点自动对焦
- 再做多点测高后的分区补偿
- 最后才考虑动态 Z 联动

难度判断：

`很高`

推荐指数：

`★★☆☆☆`

结论表达建议：

`这项能力更适合作为二期或下一代路线，而不是当前比赛版本的必做主线。`

## 4. 总体判断：哪些方向最值得做

如果从“比赛收益 / 实现风险 / 与现有工程兼容度”综合排序，我建议优先级如下：

1. `AI 主线产品化`
2. `环境与安全微基站`
3. `MQTT 遥测与可视化`
4. `BLE 手机轻控制`
5. `NFC 材质识别`
6. `RS485 测高 + 单点自动对焦`
7. `SLE 示教器`
8. `动态焦点补偿`

一句话总结：

`现在最应该做的是把现有优势放大，而不是马上跳进最重的硬件闭环。`

## 5. 分阶段推进计划

为了让计划更可执行，建议按 4 个阶段推进。

## 阶段 0：先稳住当前底座

时间建议：

- `立即开始`

目标：

- 真机验证真急停链路
- 补齐 `STA` 模式验收记录
- 继续收敛 WiFi 长时间稳定性

验收标准：

- 长轨迹中 `!` 可稳定中断
- `SoftAP` 与 `STA` 都有完整上板验证记录
- 自动化结果可沉淀成答辩材料

## 阶段 1：完成本赛季最有价值的亮点能力

时间建议：

- `第 1-2 周`

目标：

- 把 `ai_studio` 进一步比赛化
- 做出安全微基站第一版
- 明确设备状态与任务状态展示

建议交付：

- AI 任务状态面板
- 高成功率 Prompt 模板
- 安全节点联动急停演示
- 一套完整演示脚本

## 阶段 2：补多端互联与展示面

时间建议：

- `第 3-4 周`

目标：

- 选择一条轻控制入口：`BLE 手机` 或 `SLE 示教器`
- 加入 `MQTT` 遥测展示
- 视硬件条件加入 `NFC` 材质识别

建议原则：

- 不要 BLE 和 SLE 示教器两条线同时开工
- 先选一条最稳的展示路径

## 阶段 3：补感知与自动对焦

时间建议：

- `第 5-6 周`

目标：

- 接入 `RS485` 测高
- 完成单点自动对焦
- 形成“测高 -> 找焦 -> 打标”的闭环演示

注意：

- 这一阶段依赖新增硬件与较多现场联调
- 节奏应在前两阶段稳定后再推进

## 阶段 4：二期或下一代路线

时间建议：

- `比赛后 / 二期规划`

目标：

- 多点高度补偿
- `XY + Z` 联动规划
- 更复杂的动态焦点补偿
- 更强的多节点协同架构

结论：

- 这阶段更适合作为长期升级路线，而不是当前答辩必须完成的内容

---

## 6. 当前最建议马上做的 5 件事

1. 用真机验证 `!` 急停在长运动中的响应时间和恢复流程。
2. 给 `ai_studio` 增加任务状态、急停原因、已完成行数等展示信息。
3. 选定安全微基站第一版硬件：`WS63/ESP32 + 烟雾/火焰 + 风扇/蜂鸣器`。
4. 在 `BLE 手机轻控制` 和 `SLE 示教器` 之间先选一条，不要双线并行。
5. 在硬件完全定型之前，只预研“单点自动对焦”，不要直接推进动态焦点补偿。

## 7. 对外汇报时可以怎么讲

如果需要对老师、评委或队友快速说明，可以直接使用下面这段表述：

> 当前项目已经完成基于 WS63 的双板无线激光打标主链路，具备 UART/WiFi 多端接入、SLE 实时控制、接收板插补执行、DAC/PWM 输出、AI 上位机雏形、网页控制台以及真急停安全基础。  
> 后续提升方向中，AI 云边协同打标、安全微基站、MQTT 遥测展示、BLE 轻控制等方向可在当前赛季较高概率落地；NFC 材质识别、RS485 测高、Z 轴自动对焦也具备可行性，但需要新增硬件与更多联调；而无场镜振镜的 3D 动态焦点补偿难度最高，更适合作为二期或下一代升级路线。

## 8. 最终结论

这个项目现在的真实优势，不是“功能点特别散”，而是已经形成了一个很清晰的基础组合：

- `WS63 双板无线控制`
- `WiFi / SLE 多端接入`
- `AI 设计到打标的雏形闭环`
- `安全停机与工程化支撑`

因此最合理的路线，不是把所有高难度硬件都同时堆上，而是：

`先把 AI、IoT 展示、安全联动和多端控制做成完整比赛版本，再逐步进入测高、对焦和动态补偿。`
