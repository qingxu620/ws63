# WS63 项目竞赛增强版总体方案

更新时间：`2026-04-23`  
适用目录：`ws63_test/`

这份文档的目标不是推翻现有方案，而是把你们已经完成的工程底座、原有可行性计划和新增的“高分答辩构想”整合成一套更适合比赛冲刺的总方案。

核心原则只有 3 条：

1. 以当前已经跑通、已经可演示的 `WS63 双板主链路 + ai_studio` 为中心，不另起炉灶。
2. 把“AI、IoT、多端互联、安全联动”做成完整故事，优先提升比赛竞争力。
3. 对还没完全实机验收的能力，明确标为“升级路线”或“二期规划”，不虚报成熟度。

## 1. 当前最真实的项目定位

当前项目最稳妥、最有竞争力的定位，不是把它讲成一台“参数还在摸索中的实验机”，而是：

`一个基于 WS63 星闪无线链路与边缘算力的微型柔性激光打标 IoT 节点`

如果需要对外名称，建议分成两档：

### A. 当前比赛版推荐名称

《基于 WS63 星闪与边缘算力的微型柔性激光打标物联网节点》

这个名称和当前已完成能力最匹配，因为你们已经具备：

- `WS63 双板`
- `SLE 无线控制`
- `UART / WiFi 多端接入`
- `AI 生图到 G-Code` 的上位机闭环
- `真急停` 与状态回传

### B. 二期或升级版名称

《基于星闪与边缘算力的微型 3D 动态振镜打标物联网节点》

这个名称适合在以下前提满足后再正式使用：

- `RS485` 测高和自动对焦已真机打通
- `Z` 轴执行器已形成稳定闭环
- `XY + Z` 联动补偿已完成实测验证

结论：

`本赛季对外主名称建议先用“微型柔性激光打标 IoT 节点”，把 3D 动态补偿作为升级路线写进规划与答辩展望。`

## 3. 融入增强版方案后的总体架构

不改变现有主链路，只在其外层叠加“感知、展示、联动、交互”。

建议采用下面这套分层架构：

```text
用户 / 评委 / Web 大屏 / 手机 / 示教器
    ->
ai_studio 边缘设计与调度中枢
    ->
发射板 WS63
  - UART / WiFi TCP 入口
  - SLE Client
  - 状态采集 / 遥测汇聚
    ->
接收板 WS63
  - SLE Server
  - 命令队列
  - 线性插补
  - DAC 振镜输出
  - PWM 激光控制
  - 真急停与安全监控
    ->
可选扩展节点
  - 安全微基站
  - BLE 手机轻控制 或 SLE 示教器
  - 感知与对焦节点（NFC + RS485 + Z 轴）
```

这个架构最大的好处是：

- 核心控制链路不变，风险可控
- 新功能都能按模块叠加，不会把现有系统推倒重来
- 每个新模块都能单独验收、单独展示、单独降级

### 3.1 比赛版节点划分建议

结合赞助方背景、芯片约束和答辩亮点，比赛版建议明确采用：

`全 WS63 节点 + SLE 作为主亮点链路 + 安全硬件联锁保底`

不建议走两个极端：

- 不要把 `安全微基站`、`NFC`、`RS485`、`Z` 轴都拆成独立节点，节点会过多，联调复杂度会迅速上升。
- 也不要把所有扩展都塞进同一个 `WS63`，这样会让安全、感知和执行互相干扰。

当前最推荐的比赛版是 `4 节点架构`：

#### A. 发射板 `WS63`

负责：

- `UART / WiFi TCP` 接入
- 对接 `ai_studio`
- `SLE` 主控与状态汇聚
- `MQTT` 遥测上报

#### B. 接收板 `WS63`

负责：

- `XY` 振镜执行
- 激光输出
- 真急停执行
- 运行态安全监控

#### C. 安全微基站 `WS63`

负责：

- 烟雾 / 火焰检测
- 风扇 / 蜂鸣器联动
- 通过 `SLE` 或上层协议向主控发送安全告警

同时要求：

- 保留一条独立的 `硬件停光/联锁链路`
- 不把 `NFC` 或 `RS485` 这类非安全任务并进安全节点

原因：

- 安全节点职责越纯越好，便于答辩时讲“分布式物理安全”
- 即使无线链路异常，也仍有硬件级保底停光能力

#### D. 感知与对焦节点 `WS63`

负责：

- `NFC` 材料识别
- `RS485 / Modbus` 测高
- `Z` 轴执行器控制
- 单点自动对焦与标定流程

当前代码整合建议进一步明确为：

- `Z` 轴执行器第一版优先采用 `TTL UART -> 张大头 Emm_V5.0`
- `RS485 / Modbus` 优先留给测高传感器，不与当前 `Z` 轴链路混用
- 先把 `Z` 轴执行闭环做成“感知与对焦节点”的稳定底座，再逐步叠加 `NFC / 测高 / 自动对焦`

这里建议把 `NFC` 并到“感知与对焦节点”，而不是并到安全微基站。

原因：

- `NFC` 的业务语义更接近“加工前识别与工艺推荐”，不是安全告警
- `NFC` 的安装位置更接近材料托盘、测高传感器和对焦执行机构
- `NFC + 测高 + Z` 轴本身就是“加工前感知与工艺准备”这一层的统一能力

### 3.2 通信链路建议

为了同时满足“华为芯片统一性”和“星闪亮点”两点，建议这样做：

- 节点间主通信优先采用 `SLE`
- 发射板继续作为 `SLE` 主汇聚节点
- 安全节点与感知节点都优先接入 `SLE` 网络
- `BLE 手机轻控制` 与 `SLE 示教器` 二选一，避免多无线协议并发扩张

但要注意：

- `SLE` 是主亮点链路，不等于所有安全动作都只能走无线
- 对于激光停光，仍建议额外保留一条硬件联锁
- 对于 `RS485` 测高与 `Z` 轴执行，节点内部可以使用本地有线总线，不必强行二次无线化

一句话原则：

`节点之间突出 SLE，节点内部优先本地有线；安全主打“星闪联动 + 硬件保底”，感知主打“星闪组网 + 本地闭环”。`

### 3.3 节点通信关系与主从职责

为了避免后续实现时出现“谁都能控制谁”的混乱状态，建议明确采用：

`发射板单主控 + 接收板单执行 + 安全节点独立裁决 + 感知节点本地闭环`

具体关系如下：

#### A. 发射板是全局调度中心

发射板负责：

- 接收 `ai_studio`、手机或示教器的上层请求
- 统一向各节点分发任务
- 汇总接收板、安全节点、感知节点的状态
- 对外输出 `MQTT` 遥测与答辩展示数据

原则：

- 上位机不直接控制接收板
- 手机或示教器不直接控制接收板
- 感知节点和安全节点默认不直接互相通信

也就是说，正常情况下所有跨节点业务都先经过发射板。

#### B. 接收板只负责高实时执行

接收板只负责：

- `XY` 振镜运动
- 激光出光
- 真急停落地执行
- 运行态回传

它不负责：

- AI 任务管理
- `NFC` 识别
- 测高与自动对焦决策
- 环境安全判断

这样做的目的，是让接收板尽量保持“执行层纯净”，避免被外围功能拖慢。

#### C. 安全微基站负责独立安全裁决

安全微基站负责：

- 采集烟雾、火焰等安全信息
- 控制风扇、蜂鸣器等外设
- 在发现异常时立即发出安全告警

它的特殊地位在于：

- 它可以通过 `SLE` 向发射板发告警
- 它还应保留一条硬件联锁，直接参与停光

因此，安全节点不是普通状态节点，而是具备“独立否决权”的节点。

#### D. 感知与对焦节点负责加工前闭环

感知与对焦节点负责：

- 读取 `NFC`
- 执行 `RS485 / Modbus` 测高
- 控制 `Z` 轴回零、绝对移动与自动对焦
- 在本地完成滤波、标定和对焦结果判定

它的特点是：

- 对外上报的是“结果”与“状态”
- 不是把原始测高数据高频广播给所有节点

这样能避免整个系统被测高噪声和闭环细节拖入复杂同步问题。

### 3.4 建议的链路拓扑

推荐把通信拓扑理解成：

`1 条主控制链路 + 1 条安全旁路 + 若干节点内部本地总线`

```text
ai_studio / 手机 / 大屏
        |
   UART / WiFi TCP / MQTT
        |
     发射板 WS63
   (全局主控 / SLE 汇聚)
      /           \
   SLE             SLE
    /               \
接收板 WS63      安全微基站 WS63
(XY+激光执行)    (烟雾/火焰/风扇)
    \
     \ SLE
      \
   感知与对焦节点 WS63
 (NFC + RS485测高 + Z轴)

安全微基站 --硬件联锁--> 激光使能 / 接收板停光输入
感知与对焦节点 --本地总线--> NFC / RS485 传感器 / Z轴驱动
```

解释如下：

- `ai_studio -> 发射板`：走 `UART` 或 `WiFi TCP`
- `发射板 <-> 接收板`：走高优先级 `SLE` 实时控制
- `发射板 <-> 安全微基站`：走 `SLE` 安全联动
- `发射板 <-> 感知与对焦节点`：走 `SLE` 的任务级工艺控制
- `安全微基站 -> 激光`：走硬件联锁，不依赖无线
- `感知与对焦节点 -> 传感器/Z轴`：走本地有线总线，不再二次无线化

### 3.5 各链路承载什么消息

为了避免协议边界模糊，建议从第一版开始就把链路职责固定住。

#### A. `ai_studio -> 发射板`

主要承载：

- `G-Code` 任务
- 开始 / 停止 / 查询状态
- 任务参数与展示模式切换

这条链路是“上位机入口”，不是底层多节点直控链路。

#### B. `发射板 <-> 接收板`

主要承载：

- `MOVE`
- `LASER_ON / LASER_OFF`
- `SET_F / SET_S`
- `ACK`
- `STATUS`
- `ESTOP`

这是整机最关键的实时控制链路，应始终保持最小化和高优先级。

#### C. `发射板 <-> 安全微基站`

主要承载：

- `JOB_START`
- `JOB_STOP`
- `SAFE_READY`
- `ALARM_SMOKE`
- `ALARM_FIRE`
- `FAN_ON / FAN_OFF`
- `SAFE_STATE`

这条链路的本质是“安全联动事件流”，不是运动控制流。

#### D. `发射板 <-> 感知与对焦节点`

主要承载：

- `READ_NFC`
- `MEASURE_HEIGHT`
- `AUTOFOCUS`
- `HOME_Z`
- `MOVE_Z_ABS`
- `FOCUS_READY`
- 工艺参数建议

这条链路属于“加工前准备”和“工艺辅助控制”，不应与运动主链路混在一起。

当前第一版代码落地口径：

- `HOME_Z / MOVE_Z_ABS / MOVE_Z_REL / STOP_Z` 先由 `TTL UART + ZDT` 落地
- `READ_NFC / MEASURE_HEIGHT / AUTOFOCUS` 先保留为节点协议与扩展点

### 3.6 正常工作时的时序建议

建议把一次完整任务拆成下面的顺序：

1. `ai_studio` 把任务发送给发射板。
2. 发射板先请求感知节点执行 `NFC` 识别、测高与自动对焦。
3. 感知节点返回材料类型、高度值、推荐参数、`Z` 轴到位结果。
4. 发射板确认工艺参数后，通知安全微基站进入加工准备状态，例如开风扇。
5. 安全微基站返回 `SAFE_READY`。
6. 发射板开始通过 `SLE` 向接收板下发轨迹和功率控制。
7. 接收板持续返回 `ACK / STATUS / 位置 / 运行态`。
8. 发射板把状态同步给 `ai_studio` 与 `MQTT` 大屏。
9. 加工结束后，发射板通知安全微基站进入延时排烟，再停止整机。

这一顺序的核心是：

`先感知与工艺准备，再安全确认，最后才进入运动执行。`

### 3.7 异常与急停时的优先级关系

这部分必须提前定清楚，否则后面越做越乱。

#### A. 安全异常优先级最高

一旦安全微基站检测到明火、严重烟雾或其他危险状态，应同时执行两件事：

1. 通过 `SLE` 向发射板发送 `ALARM`
2. 通过硬件联锁直接切断激光使能或拉起停光信号

也就是说：

`真正最先起作用的应是硬件联锁，其次才是无线急停广播。`

#### B. 发射板负责全局收口

发射板收到安全告警后，应立即：

- 停止继续下发任务
- 向接收板广播 `ESTOP`
- 将异常状态同步到 `ai_studio`
- 将异常原因同步到 `MQTT` 大屏与日志

#### C. 接收板负责最终执行落地

接收板收到 `ESTOP` 或本地安全条件不满足时，应立即：

- 关闭激光
- 清空队列
- 打断当前运动
- 回传当前急停状态

#### D. 感知节点异常不应直接控制整机运动

例如：

- 测高失败
- `NFC` 读取失败
- `Z` 轴未到位

这类异常应先上报给发射板，由发射板决定：

- 阻止任务开始
- 提示人工介入
- 切换到手动模式

而不是让感知节点越级直接控制运动主链路。

### 3.8 协议设计建议

建议在 `SLE` 应用层预留统一消息结构，至少包含：

- `src_id`
- `dst_id`
- `msg_type`
- `seq`
- `priority`
- `payload`
- `crc`

并建议至少定义 3 个优先级：

- `P0`：`ESTOP / ALARM / SAFE_FAULT`
- `P1`：`MOVE / ACK / STATUS`
- `P2`：`NFC / 测高 / 对焦 / 遥测`

规则：

- `P0` 永远可以打断 `P1 / P2`
- `P1` 不应被 `P2` 长时间阻塞
- `P2` 允许重试和排队，但不应影响急停与运动主链路

### 3.9 最终通信原则

建议把整套关系收敛成下面 5 条原则：

1. 全局控制统一经发射板中转，不做多主控并发控制。
2. 接收板只管实时执行，不承载感知与展示逻辑。
3. 安全节点既走 `SLE` 联动，也保留硬件联锁。
4. 感知节点本地完成 `NFC + 测高 + Z` 轴闭环，再把结果上报给发射板。
5. 节点之间突出 `SLE`，节点内部优先本地有线。

| 测高与自动对焦 | 现有主控链路与安全控制框架 | 增加 `RS485 / Modbus`、滤波、标定、Z 轴协议 | 形成技术高地 |
| 动态 Z 补偿 | 当前为概念预留，不是现成能力 | 后续扩协议、扩插补、扩标定模型 | 作为二期突破点 |

## 5. 对原计划的关键调整

这次整合不是“什么都做”，而是按比赛收益重新排序。

### 5.1 保留并放大的部分

必须继续强化：

- `AI 生图 -> 轮廓 -> G-Code -> 下发` 主闭环
- `SoftAP / STA` 双工作流
- 紧凑状态面板、急停反馈、已完成行数展示
- 历史任务复用和日志归档
- 真急停与安全链路

### 5.2 明确降级或延后的部分

为了聚焦产出，建议这样处理：

- `Prompt 模板系统` 不再单独投入开发
- 不把“完整网站后端”作为本赛季前置条件
- 不把 `3D 动态焦点补偿` 当作本赛季必须完成项

替代方案：

- 用 `历史成功任务复用 + 演示素材库 + 一键美化 Prompt` 替代模板系统
- 用 `ai_studio` 作为稳定主入口，Web 只做展示或轻入口
- 用 `单点自动对焦 / 测高演示` 代替过重的实时动态补偿承诺

这一调整的意义是：

`把有限时间集中到最能提升分数、最能稳定演示、最容易讲清楚的模块上。`

## 6. 竞赛版能力包建议

建议把整套方案拆成 `P0 / P1 / P2` 三档。

## 6.1 P0：本赛季必须交付的比赛版

这部分是你们当前框架最适合完成、收益也最高的能力。

### A. AI 柔性制造闭环

目标：

- 评委输入一句 Prompt，或直接复用历史成功任务
- `ai_studio` 自动完成生图、轮廓提取、G-Code 生成、设备下发
- 界面全程显示任务状态、执行进度和结果归档

当前基础：

- 已基本具备，只需继续打磨演示体验

### B. 环境与安全微基站

目标：

- 增加独立安全节点
- 支持烟雾/火焰检测
- 加工时联动风扇或蜂鸣器
- 异常时向主控发告警并触发急停

当前基础：

- 真急停链路已具备
- 只差独立节点与联动协议

### C. MQTT 数字孪生大屏

目标：

- 把设备状态、任务状态、急停原因、WiFi/SLE 状态、坐标和进度上报到大屏
- 形成“云端设备影子”展示

当前基础：

- 状态快照和任务状态机已经有了
- 新增一个发布器和一个可视化前端即可

### D. 演示样例库与历史复用

目标：

- 准备若干稳定成功案例
- 现场可快速复用，无需每次都赌 AI 出图
- 支持“离线保底演示”和“在线增强演示”两条路径

当前基础：

- 历史任务、产物归档、日志已经具备

结论：

`P0 做完后，你们的作品已经不是单机演示，而是一个“AI 设计 + 边缘执行 + IoT 展示 + 安全联动”的完整赛题答案。`

## 6.2 P1：高性价比加分项

这部分适合在 P0 稳定后再加。

### A. 手机轻控制 或 SLE 示教器

原则：

- 两条线只选一条
- 不要同时开工

建议：

- 如果目标是更快落地，优先 `BLE 手机轻控制`
- 如果目标是突出芯片协议能力，优先 `SLE 示教器`

最小功能：

- 查看状态
- 开始 / 停止

- 简单 `jog`

### B. NFC 材质识别与工艺推荐

- 读取材料 UID

## 6.3 P2：技术高地与二期路线

这部分写进计划书和答辩展望，但不建议拿来卡死本赛季验收。

### A. RS485 测高 + 单点自动对焦

不要在未实测前对外宣称：

- 已完成高频动态跟焦

## 12. 最终判断

这次整合之后，项目最合理的竞争策略已经很明确：

- 核心主线继续围绕 `WS63 双板 + SLE + ai_studio`
- 竞赛亮点优先补 `AI 闭环、IoT 大屏、安全联动、多端控制`
- 高难度机械闭环不从叙事中删掉，但从“当前必须完成”降级为“技术高地路线”

一句话总结就是：

`先把作品做成一个完整、可信、可演示、可讲清楚的微型柔性制造 IoT 系统，再把自动对焦和动态补偿作为后续突破点逐步抬高上限。`
