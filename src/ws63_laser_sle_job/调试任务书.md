# WS63 SLE Job 无线离线打标调试任务书

本文档用于调试 `ws63_laser_sle_job` 新主线工程。

目标不是兼容 LaserGRBL，也不是继续修 SLE 透明串口，而是把问题暴露在发射板 COM24 和接收板 COM26 日志中，按证据定位。

## 1. 系统目标

系统结构：

```text
PC 自建上位机
  -> USB 串口
TX 发射板
  -> SLE 标准任务包
RX 接收板
  -> RAM 任务缓存
  -> G-code 本地解析
  -> motion queue
  -> DAC / 激光控制
```

第一阶段验收目标：

1. PC 能把完整 G-code 文件下发给 TX。
2. TX 能封装为 `JOB_BEGIN / JOB_DATA / JOB_END` 标准包。
3. RX 能按 `seq / offset / crc16` 完整接收并校验。
4. RX 收到 `EXEC_START` 后从本地缓存执行。
5. 执行完成后激光强制关闭。
6. `EXEC_STOP / JOB_ABORT / SLE 断连` 必须立即关光并清空运动队列。

## 2. 串口职责

### COM24：TX 发射板调试口

COM24 只看 TX 的链路行为：

- 是否收到上位机命令。
- 是否进入数据接收模式。
- 是否发送 SLE 包。
- 是否收到 RX 的 ACK/NACK。
- 是否出现超时、重发、失败。

重点日志：

```text
[JOB_TX] uart init ...
[tx] ready!
[JOB_TX_FRAME] type=... seq=... len=... try=... ret=...
[JOB_TX_ACK] ack_type=... ack_seq=... status=... job=... off=... credit=...
[JOB_TX_STATUS] state=... status=... job=... rx=... total=...
```

### COM26：RX 接收板调试口

COM26 只看 RX 的业务状态：

- 是否收到 SLE 标准包。
- 是否解包 CRC 正确。
- 是否 seq 正确。
- 是否 offset 正确。
- 是否任务缓存完整。
- 是否进入执行状态。
- 是否安全停机。

重点日志：

```text
[JOB_RX] manager init cache=...
[job_rx] advertising as 'sle_job_rx'
[job_rx] SLE connected conn_id=...
[JOB_BEGIN] seq=... job=... size=... crc=... st=...
[JOB_DATA] seq=... job=... off=... len=... rx=... st=...
[JOB_END] seq=... job=... size=... crc=... st=... state=JOB_READY
[EXEC_START] seq=... job=... st=... state=...
[JOB_EXEC] start job=... size=...
[JOB_EXEC] done job=... lines=... x=... y=...
[JOB_SAFE_STOP] reason=...
```

## 3. 状态码定义

来自 `common/protocol.h`：

```text
0  JOB_STATUS_OK
1  JOB_STATUS_BAD_MAGIC
2  JOB_STATUS_BAD_CRC
3  JOB_STATUS_BAD_SEQ
4  JOB_STATUS_BAD_OFFSET
5  JOB_STATUS_BAD_STATE
6  JOB_STATUS_NO_SPACE
7  JOB_STATUS_BAD_JOB
8  JOB_STATUS_NOT_READY
9  JOB_STATUS_INTERNAL_ERROR
```

调试时，任何 `status != 0` 都必须保存 COM24 和 COM26 日志。

## 4. 第一轮基础连通测试

### 4.1 上电顺序

建议顺序：

1. RX 接收板上电。
2. TX 发射板上电。
3. 等待 COM26 出现：

```text
[job_rx] advertising as 'sle_job_rx'
```

4. 等待 COM24 出现：

```text
[tx] ready!
```

### 4.2 验收标准

COM24 必须出现：

```text
[tx] found receiver
[tx] connected
[tx] ready!
```

COM26 必须出现：

```text
[job_rx] SLE connected conn_id=...
```

如果 TX ready 没出现，优先看 SLE 广播名、UUID、连接参数。

## 5. 第二轮 STATUS 测试

PC 发：

```text
@STATUS
```

TX 应通过 COM24 打印：

```text
[JOB_TX_STATUS] state=0 status=0 job=0 rx=0/0 free=... lines=0
```

PC 串口应收到：

```text
@STATUS state=0 status=0 job=0 rx=0 total=0 free=... lines=0
```

状态解释：

```text
state=0 表示 IDLE
status=0 表示 OK
```

如果 COM24 没有 STATUS：

- 看 `[JOB_TX_FRAME] type=0x20` 是否发出。
- 看 COM26 是否收到 `PKT_STATUS_REQ`。
- 看 COM26 是否发送 `STATUS_RESP`。

## 6. 第三轮最小任务下发测试

### 6.1 G-code 文件内容

建议先用最小文件：

```gcode
G90
M3 S50
G1 X10 Y10 F1000
M5
```

注意：CRC16 必须按原始文件字节计算，包括换行符。

### 6.2 PC 发送流程

PC 发送：

```text
@BEGIN 1 <total_size> <crc16_hex>
```

等待：

```text
@DATA_READY job=1 size=<total_size>
```

然后发送精确 `total_size` 字节 G-code 文件内容。

等待：

```text
@JOB_READY job=1 size=<total_size>
```

最后发送：

```text
@EXEC_START 1
```

### 6.3 COM24 期望日志

必须看到：

```text
[JOB_TX_FRAME] type=0x01 seq=... len=... try=0 ret=0x0
[JOB_TX_ACK] ack_type=0x01 ack_seq=... status=0 job=1 off=0 credit=...

[JOB_TX_FRAME] type=0x02 seq=... len=... try=0 ret=0x0
[JOB_TX_ACK] ack_type=0x02 ack_seq=... status=0 job=1 off=... credit=...

[JOB_TX_FRAME] type=0x03 seq=... len=... try=0 ret=0x0
[JOB_TX_ACK] ack_type=0x03 ack_seq=... status=0 job=1 off=<total_size> credit=...

[JOB_TX_FRAME] type=0x10 seq=... len=... try=0 ret=0x0
[JOB_TX_ACK] ack_type=0x10 ack_seq=... status=0 job=1 off=<total_size> credit=...
```

### 6.4 COM26 期望日志

必须看到：

```text
[JOB_BEGIN] seq=... job=1 size=<total_size> crc=0x.... st=0
[JOB_DATA] seq=... job=1 off=0 len=... rx=... st=0
[JOB_END] seq=... job=1 size=<total_size> crc=0x.... st=0 state=JOB_READY
[EXEC_START] seq=... job=1 st=0 state=JOB_READY
[JOB_EXEC] start job=1 size=<total_size>
[JOB_EXEC] done job=1 lines=... x=... y=...
```

执行结束后，激光必须关闭。

## 7. 第四轮异常注入测试

这些测试用于确认 bug 能在 COM24/COM26 中暴露出来。

### 7.1 CRC 错误

故意发送错误 CRC：

```text
@BEGIN 2 <total_size> 0001
```

期望：

COM26：

```text
[JOB_END] ... st=2 state=ERROR
```

COM24：

```text
[JOB_TX_ACK] ... status=2 ...
```

结论：

CRC 错误能被 RX 发现，TX 能收到 NACK。

### 7.2 offset 错误

第一版 TX 正常不会产生 offset 错误。如果后续自建上位机直接发二进制包，可以故意跳 offset。

期望：

COM26：

```text
[JOB_CACHE] bad offset job=... got=... expect=...
[JOB_DATA] ... st=4
```

COM24：

```text
[JOB_TX_ACK] ... status=4 ...
```

### 7.3 seq 错误

故意重发旧 seq 或跳 seq。

期望：

COM26：

```text
[JOB_RX] seq error type=... got=... expect=...
```

COM24：

```text
[JOB_TX_ACK] ... status=3 ...
```

### 7.4 任务未完整就执行

不要发送完整文件，直接发：

```text
@EXEC_START 1
```

期望：

COM26：

```text
[EXEC_START] ... st=8
```

COM24：

```text
[JOB_TX_ACK] ... status=8 ...
```

## 8. 第五轮安全停机测试

### 8.1 执行中停止

准备一个较长 G-code 文件，执行后立刻发送：

```text
@EXEC_STOP
```

期望 COM26：

```text
[JOB_SAFE_STOP] reason=exec-stop state=EXECUTING
```

实际机械状态：

- 激光立即关闭。
- motion queue 清空。
- 后续不再继续执行旧任务。

### 8.2 任务中止

发送：

```text
@ABORT
```

期望 COM26：

```text
[JOB_SAFE_STOP] reason=job-abort ...
```

并且缓存清空，状态回到 IDLE。

### 8.3 SLE 断连

执行中手动复位 TX 或让 TX 断电。

期望 COM26：

```text
[job_rx] SLE disconnected, force safe stop
[JOB_SAFE_STOP] reason=sle-disconnect ...
```

实际机械状态：

- 激光必须立即关闭。
- 队列必须停止。

## 9. 日志定位规则

### 9.1 PC 发了命令但 TX 没反应

看 COM24：

- 是否出现 `@ERR`。
- 是否出现 `[JOB_TX_FRAME]`。

如果没有 `[JOB_TX_FRAME]`，问题在 PC->TX UART 协议。

### 9.2 TX 发包了但 RX 没收到

COM24 有：

```text
[JOB_TX_FRAME]
```

COM26 没有：

```text
[JOB_BEGIN] / [JOB_DATA] / [JOB_END]
```

问题在 SLE 发送、连接、handle、UUID、write confirm。

### 9.3 RX 收到包但 TX 没 ACK

COM26 有：

```text
[JOB_DATA] ... st=0
```

COM24 没有：

```text
[JOB_TX_ACK]
```

问题在 RX notify 回传、TX response callback、ACK 包 decode。

### 9.4 RX NACK

看 `status`：

```text
2 -> CRC 错
3 -> seq 错
4 -> offset 错
5 -> 状态不允许
6 -> 缓存不够
8 -> 任务未 ready
```

### 9.5 任务 ready 但不执行

COM26 必须有：

```text
[EXEC_START]
```

如果 `st=8`，说明：

- `JOB_END` 没成功。
- CRC 没通过。
- job_id 不匹配。

如果有 `[JOB_EXEC] start` 但没有运动：

- 看 `gcode_process_line` 是否解析失败。
- 看 `[JOB_EXEC] parse fail`。
- 看 `[JOB_EXEC] enqueue fail`。

## 10. 每次反馈必须提供的信息

后续反馈 bug 时，请至少提供：

1. 当前烧录的是 TX 还是 RX。
2. 当前 menuconfig 角色。
3. PC 发出的完整命令序列。
4. COM24 从开始发送到异常后的完整日志。
5. COM26 从开始发送到异常后的完整日志。
6. 是否有实际出光、运动、停光异常。

没有 COM24/COM26 日志时，不进行代码推测式修改。

## 11. 当前第一版已知边界

1. RX 任务缓存默认 128KB。
2. `JOB_DATA` 第一版只支持顺序 offset，不支持乱序重排。
3. 第一版不支持边传边打。
4. 第一版不支持断点续传。
5. 第一版不做高速窗口流控。
6. 第一版目标是稳定下发、完整校验、本地执行、安全停机。

## 12. 编译和烧录提醒

默认配置现在是 RX：

```text
CONFIG_ENABLE_LASER_SLE_JOB_SAMPLE=y
CONFIG_LASER_SLE_JOB_RECEIVER=y
```

所以直接编译得到的是接收板固件：

```bash
python3 build.py -c ws63-liteos-app
```

烧录到 RX 接收板。

要编译 TX：

```bash
python3 build.py -c ws63-liteos-app menuconfig
```

选择：

```text
WS63 Laser SLE Job Configuration
  -> Build TRANSMITTER firmware
```

再编译，并烧录到 TX 发射板。

同一个输出路径：

```text
output/ws63/fwpkg/ws63-liteos-app/ws63-liteos-app_all.fwpkg
```

代表“当前 menuconfig 角色”的固件，不要混烧。
