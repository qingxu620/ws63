# WS63 赛题两个月冲刺方案

这份文档不是泛泛而谈的“比赛建议”，而是基于当前 `ws63_test` 工程现状给出的冲刺版落地方案。

当前作品已经具备较好的基础：

- 主控满足赛题要求，基于 `WS63 + LiteOS`
- 已完成双板分工，形成 `上位机 -> 发射板 -> SLE -> 接收板` 闭环
- 已具备 `SLE + WiFi + UART + AI 上位机` 的可演示链路
- 已形成自动化回归脚本和压测基线

但如果目标是“比赛获奖作品”，还需要把“能跑”继续拉升到“题目贴合度高、演示有记忆点、现场稳定、答辩可量化”。

## 1. 现状判断

按赛题要求拆解，你们当前工程的匹配度如下：

| 赛题点 | 当前状态 | 结论 |
| --- | --- | --- |
| 主控使用 WS63 / WS63E / BS21 | 已使用 WS63 | 已满足 |
| 操作系统 LiteOS / OpenHarmony | 当前为 LiteOS | 已满足 |
| 嵌入式系统开发能力 | 双板固件、插补、安全、协议、驱动已具备 | 已满足 |
| SLE 多端互联 | 发射板 Client + 接收板 Server 已完成 | 已满足 |
| WiFi 多端互联 | 发射板已有 SoftAP/STA + TCP 入口 | 已满足 |
| BLE 多端互联 | 当前作品内未形成明确 BLE 业务入口 | 当前最大短板 |
| 作品完整度 | 已有 AI 上位机原型、自动化测试、文档 | 中上 |
| 现场展示力 | 可演示，但缺少“手机/BLE 控制”和“可视化闭环” | 需加强 |

一句话判断：

你们现在已经有“可交付工程样机”，但距离“比赛强作品”还差一块最关键的拼图：`BLE 入口 + 更强的作品故事线 + 更直观的可视化展示`。

## 2. 最该补的三件事

### 第一优先级：补齐 BLE 端能力

这是你们最应该优先完成的方向，因为它直接对应赛题里的 `SLE/BLE/WiFi 多端互联能力`。

建议做法：

- 在发射板新增 `BLE GATT Server`
- 让手机小程序/蓝牙调试助手可连接发射板
- 提供最小但完整的 BLE 特征值：
  - `控制指令写入`
  - `设备状态通知`
  - `网络状态读取`
  - `任务控制/急停`
- BLE 不必一开始就承载完整 G-Code 流
- 比赛版本更建议 BLE 承担“轻控制 + 状态查看 + 一键演示”角色

这样做的好处：

- 赛题匹配度立刻明显提升
- 演示形式从“电脑串口工具”升级为“手机 + WiFi + 星闪双板”
- 复杂度可控，不会破坏现有 UART/WiFi 主链路

代码落点建议：

- 新增 `ws63_test/transmitter/ble_control_server.c`
- 新增 `ws63_test/transmitter/ble_control_server.h`
- 在 [main.c](/root/fbb_ws63/src/ws63_test/transmitter/main.c) 中增加 BLE 初始化任务
- 复用 [wifi_gcode_server_get_status()](/root/fbb_ws63/src/ws63_test/transmitter/wifi_gcode_server.h) 与 `sle_client` 状态快照能力

可参考仓库现成示例：

- `application/samples/bt/ble/`

### 第二优先级：把“比赛作品故事”做完整

你们现在的技术链路没问题，但作品名称和展示逻辑还偏“工程调通”。比赛答辩更吃“完整方案”。

建议把作品重新包装为：

`基于 WS63 星闪双板协同与 WiFi/BLE 多端互联的 AI 智能激光打标系统`

建议把现场演示固定为三幕：

1. 手机 BLE 连接发射板，查看设备状态并选择模板
2. 电脑或网页端通过 WiFi 下发图案/G-Code
3. 发射板通过 SLE 向接收板实时下发控制，接收板完成激光打标并回传状态

这样答辩时会非常顺：

- BLE 体现“近端控制”
- WiFi 体现“网络接入/多端内容下发”
- SLE 体现“板间低时延可靠控制”

### 第三优先级：把稳定性做成“有数据的稳定性”

你们已经有 `stress_test.py`，这很宝贵，下一步不是“再说稳定”，而是做“稳定性证据”。

建议补三类量化指标：

- 建链时间：发射板上电到 SLE Ready 的平均时间
- 指令时延：上位机发命令到接收板 ACK 的平均/最大时延
- 压测结果：`50/50 PASS` 之外再补更长时长或更复杂轨迹

建议形成一页答辩表格：

- `SLE 建链成功率`
- `WiFi 入网成功率`
- `连续 100 次图形任务通过率`
- `异常断链后激光关闭时间`

## 3. 两个月排期建议

### 第 1-2 周：补 BLE 最小可用版本

目标：

- 手机能连 BLE
- 能读到设备状态
- 能发送开始/停止/急停/模式切换等轻量命令
- 不改现有 SLE 主链路

验收标准：

- 手机可看到发射板
- 可读出 `SLE/WiFi/队列/连接` 状态
- 可触发急停与恢复

### 第 3-4 周：做演示闭环和 UI

目标：

- 增加网页端或简单手机控制页
- 统一作品对外命令语义
- 固定 2-3 套演示素材

验收标准：

- 评委不需要串口工具也能看懂系统怎么用
- BLE/WiFi/SLE 三条链路在一次演示中都能出现

### 第 5-6 周：做安全与异常恢复

目标：

- 增加更强的异常保护逻辑
- 完善断链、停光、队列清空、复位恢复
- 如果硬件允许，增加物理急停按钮或限位/仓门检测

建议补的安全点：

- BLE 急停命令
- WiFi 断开后的任务处理策略
- 接收板异常状态码细化
- 错误事件上报给上位机/手机

### 第 7 周：压测和答辩材料固化

目标：

- 跑完整回归
- 整理测试数据
- 固化演示脚本

建议输出：

- 1 页系统架构图
- 1 页通信分工图
- 1 页稳定性测试结果
- 1 页创新点总结

### 第 8 周：只做收敛，不再大改

最后一周原则：

- 不换架构
- 不大改协议
- 不新引入高风险功能
- 只修现场能看出来的问题

## 4. 建议新增的比赛亮点

如果精力允许，下面这些功能会明显加分：

### 方案 A：手机 BLE 一键模板雕刻

最稳、最适合比赛。

做法：

- 手机 BLE 选择预置图案
- 发射板从本地模板表生成简单轨迹
- 通过 SLE 下发给接收板执行

优点：

- 演示独立，不依赖电脑和网络环境
- 评委能直观看到 BLE 的价值

### 方案 B：WiFi 网页端实时预览

做法：

- 发射板开 SoftAP
- 手机或电脑打开网页
- 上传图案或点击模板
- 页面显示设备在线状态与任务进度

优点：

- 展示效果更现代
- 更容易体现“物联网作品”

### 方案 C：状态大屏/日志看板

做法：

- PC 端显示 `WiFi 状态 / SLE 状态 / 当前坐标 / 队列余量 / 激光状态`
- 把底层状态变成可视化

优点：

- 评委更容易理解你们的系统不是“黑盒”
- 可以直接展示你们的工程能力

## 5. 当前不建议做的事

为了保证两个月内成功率，这些方向不建议优先投入：

- 彻底重写现有协议
- 追求复杂图像算法替代当前可用 AI/G-Code 流程
- 同时支持多个上游并发控制
- 在最后阶段切 OpenHarmony 重构整套工程

原因很简单：

你们现在最值钱的是“已经跑通的稳定主链路”，后续改进应该围绕它增强，而不是推翻它。

## 6. 我对你们当前作品的建议结论

如果只给一个冲刺方向，我建议你们这样定：

`保住现有 SLE 激光控制主链路，优先补 BLE 轻控制入口，再用 WiFi 做内容下发和可视化展示。`

这样做可以同时满足：

- 赛题要求
- 两个月可落地
- 现场展示效果
- 答辩逻辑清晰
